# Регистры SCR1: MPRF и CSR

> Источники: `scr1/src/includes/scr1_arch_types.svh` (размеры MPRF), `scr1/src/core/pipeline/scr1_pipe_mprf.sv` (регистровый файл), `scr1/src/includes/scr1_csr.svh` (карта CSR), `scr1/src/core/pipeline/scr1_pipe_csr.sv` (реализация CSR).
> Документация: `scr1_eas.pdf` §3–4 (регистры, CSR), спецификация RISC-V Privileged.

Дополнение к разборам конвейера ([IFU](scr1_ifu_explained.md), [IDU](scr1_idu_explained.md), [EXU](scr1_exu_explained.md)). В процессоре есть **два совершенно разных «регистровых файла»**, и их важно не путать:

- **MPRF** (Multi-Port Register File) — 32 регистра общего назначения `x0…x31`, с которыми работает почти каждая инструкция (операнды `rs1`/`rs2`, результат `rd`).
- **CSR** (Control and Status Registers) — управляющие/статусные регистры процессора в отдельном адресном пространстве, доступные только специальными инструкциями `CSR*`.

Всё ниже — для **нашей конфигурации**: RV32IMC (RVE выключен), target Xilinx/Nexys4DDR, с включённой отладкой (DBG/TDU) и контроллером прерываний (IPIC).

---

# 1. MPRF — регистры общего назначения

## 1.1. Размерность и формат

Из `scr1_arch_types.svh` (ветка при выключенном RVE):

```systemverilog
`define SCR1_MPRF_AWIDTH  5    // 5-битный адрес регистра
`define SCR1_MPRF_SIZE    32   // 32 регистра
```

- **32 регистра** `x0`–`x31`, каждый шириной **32 бита** (`SCR1_XLEN = 32`).
- Формат — **плоский**: просто 32-битное целое, **никаких под-полей**. Это обычные ячейки для данных и адресов.
- **`x0` жёстко равен нулю**: запись игнорируется, чтение всегда возвращает 0. В RTL это видно по тому, что массив объявлен `[1:31]` (нулевого регистра физически нет), а чтение по адресу 0 отдаёт `'0`:

```systemverilog
// scr1_pipe_mprf.sv (распределённая логика, наша сборка)
assign mprf2exu_rs1_data_o = (rs1_addr_vd) ? mprf_int[exu2mprf_rs1_addr_i] : '0;  // addr==0 → 0
assign mprf2exu_rs2_data_o = (rs2_addr_vd) ? mprf_int[exu2mprf_rs2_addr_i] : '0;
```

## 1.2. Реализация в нашей сборке

MPRF бывает двух видов; выбор — по `SCR1_MPRF_RAM`, который включается **только для Intel FPGA**:

```systemverilog
// scr1_arch_description.svh
`ifdef SCR1_TRGT_FPGA_INTEL
  `define SCR1_MPRF_RAM      // MPRF на блочной RAM (синхронное чтение)
`endif
```

Наш target — Xilinx, поэтому `SCR1_MPRF_RAM` **не задан** → MPRF на **распределённой логике с асинхронным (комбинационным) чтением** и сбросом (`SCR1_MPRF_RST_EN`). Асинхронное чтение — причина, по которой операнды доезжают до АЛУ в том же такте (см. [scr1_exu_explained.md](scr1_exu_explained.md) §2).

> В MPRF есть свой bypass — обход коллизии «запись и чтение одного регистра в одном такте» (read-during-write). Он не связан с bypass в IFU; подробнее — врезка в [scr1_idu_explained.md](scr1_idu_explained.md) §4.

## 1.3. Роли регистров (RISC-V ABI)

Физически регистры безымянные (`x0…x31`), но по стандартному соглашению о вызовах у них есть роли:

| Рег. | ABI | Назначение | Сохраняет |
|---|---|---|---|
| `x0` | `zero` | всегда 0 | — |
| `x1` | `ra` | адрес возврата | caller |
| `x2` | `sp` | указатель стека | callee |
| `x3` | `gp` | global pointer | — |
| `x4` | `tp` | thread pointer | — |
| `x5`–`x7` | `t0`–`t2` | временные | caller |
| `x8` | `s0`/`fp` | сохраняемый / frame pointer | callee |
| `x9` | `s1` | сохраняемый | callee |
| `x10`–`x11` | `a0`–`a1` | аргументы / возвращаемое значение | caller |
| `x12`–`x17` | `a2`–`a7` | аргументы | caller |
| `x18`–`x27` | `s2`–`s11` | сохраняемые | callee |
| `x28`–`x31` | `t3`–`t6` | временные | caller |

> `sp`(x2) и `ra`(x1) «зашиты» в декодере сжатых команд (C.LWSP, C.SWSP, C.JAL, C.JALR и т.д.) — см. [scr1_idu_explained.md](scr1_idu_explained.md) §8.

# 2. CSR — Control and Status Registers

## 2.1. Общее

- Отдельное адресное пространство: **12-битный адрес**, значение **32-битное**.
- Доступ — только инструкциями `CSRRW / CSRRS / CSRRC / CSRRWI / CSRRSI / CSRRCI`. SCR1 обслуживает их в EXU (см. [scr1_exu_explained.md](scr1_exu_explained.md) §8).
- Старшие биты адреса кодируют доступ: `[11:10] = 11` → **read-only**; привилегия — в `[9:8]` (`11` = machine).
- SCR1 реализует **только M-режим** (machine mode).

Ниже — все CSR из `scr1_csr.svh`, активные в нашей конфигурации.

## 2.2. Machine Information (read-only)

| Адрес | Имя | Значение в нашей сборке |
|---|---|---|
| `0xF11` | `mvendorid` | `0x0000_0000` |
| `0xF12` | `marchid` | `0x0000_0008` (идентификатор SCR1 = 8) |
| `0xF13` | `mimpid` | `0x2201_1200` (build ID) |
| `0xF14` | `mhartid` | ID харта (0 для одиночного ядра) |

## 2.3. Machine Trap Setup (read-write)

| Адрес | Имя | Формат / поля |
|---|---|---|
| `0x300` | `mstatus` | **MIE** (бит 3) — глобальное разрешение прерываний; **MPIE** (бит 7) — прежнее значение MIE; **MPP** (биты 12:11) — всегда `11` (M). Остальные биты — 0 |
| `0x301` | `misa` | **MXL** (старшие 2 бита) = `01` (32 бита) + буквы расширений. У нас `0x4000_1104` = **RV32 I·M·C**: I (бит 8), M (бит 12), C (бит 2) |
| `0x304` | `mie` | Маски разрешения: **MSIE** (бит 3, software), **MTIE** (бит 7, timer), **MEIE** (бит 11, external) |
| `0x305` | `mtvec` | Адрес обработчика трапов: **BASE** (биты 31:6, выровнен на 64 байта) + **MODE** (0 = direct, 1 = vectored) |

Кодировка `misa` собирается препроцессором прямо из включённых расширений:

```systemverilog
// scr1_csr.svh
`define SCR1_RVC_ENC  `SCR1_XLEN'h0004   // C → бит 2
`define SCR1_RVI_ENC  `SCR1_XLEN'h0100   // I → бит 8
`define SCR1_RVM_ENC  `SCR1_XLEN'h1000   // M → бит 12
parameter bit [`SCR1_XLEN-1:0] SCR1_CSR_MISA = (SCR1_MISA_MXL_32 << (`SCR1_XLEN-2))
                                             | `SCR1_RVI_ENC
                                             | `SCR1_RVC_ENC     // при SCR1_RVC_EXT
                                             | `SCR1_RVM_ENC;    // при SCR1_RVM_EXT
```

## 2.4. Machine Trap Handling (read-write)

| Адрес | Имя | Формат |
|---|---|---|
| `0x340` | `mscratch` | Свободный 32-битный регистр для нужд обработчика |
| `0x341` | `mepc` | PC, сохранённый при трапе (бит 0 = 0) |
| `0x342` | `mcause` | **бит 31** = 1 для прерывания / 0 для исключения; **биты 30:0** — код причины |
| `0x343` | `mtval` | Trap value: адрес при ошибке памяти, тело нелегальной инструкции и т.п. |
| `0x344` | `mip` | Ожидающие прерывания: **MSIP** (бит 3), **MTIP** (бит 7), **MEIP** (бит 11) |

## 2.5. Счётчики (в нашей сборке — полные 64-битные)

| Адрес | Имя (RW) | | Адрес | Имя (RO shadow) |
|---|---|---|---|---|
| `0xB00` | `mcycle` | | `0xC00` | `cycle` |
| `0xB80` | `mcycleh` | | `0xC80` | `cycleh` |
| `0xB02` | `minstret` | | `0xC02` | `instret` |
| `0xB82` | `minstreth` | | `0xC82` | `instreth` |
| | | | `0xC01` | `time` (+ `0xC81` `timeh`) |

Счётчики 64-битные — отсюда пары `…` / `…H` (младшая / старшая половины). `time` — read-only тень внешнего таймера. (При выключенном RVE счётчики полные; `SCR1_CSR_REDUCED_CNT` включился бы только для RVE.)

## 2.6. Нестандартные (Syntacore-специфичные) CSR

| Адрес | Имя | Что это |
|---|---|---|
| `0x7E0` | `mcounten` | Кастомный контроль счётчиков: **CY** (бит 0 → mcycle), **IR** (бит 2 → minstret). Включён `SCR1_MCOUNTEN_EN` |
| `0xBF0` + off | `ipic_*` | Регистры **IPIC** (интегрированного контроллера прерываний): CISV, CICSR, IPR, ISVR, EOI, SOI, IDX, ICSR |
| `0x7A0` (base) | `tdu_*` | **TDU** — аппаратные точки останова (2 триггера), span 8 |
| `0x7B0` (base) | `hdu_*` | **HDU** — отладочная подсистема, span 4 |

# 3. Итог

- **MPRF** — 32 плоских 32-битных регистра `x0…x31` (x0 = 0), без под-полей; обычные данные и адреса, читаются/пишутся почти каждой инструкцией через EXU. В нашей сборке — распределённая логика с асинхронным чтением.
- **CSR** — отдельное 12-битное адресное пространство 32-битных управляющих регистров **M-режима**: информация о ядре (`mvendorid`/`marchid`/`mimpid`/`mhartid`), настройка трапов (`mstatus`/`misa`/`mie`/`mtvec`), обработка трапов (`mepc`/`mcause`/`mtval`/`mip`/`mscratch`), счётчики (`mcycle`/`minstret` + read-only тени), плюс нестандартные `mcounten`, IPIC, TDU, HDU. Доступ — только инструкциями `CSR*`.

Логичное продолжение — разбор модуля `scr1_pipe_csr.sv`: как эти CSR физически читаются/пишутся, как формируется трап (запись `mepc`/`mcause`/`mtval`, переход по `mtvec`), и как считается приоритет прерываний.
