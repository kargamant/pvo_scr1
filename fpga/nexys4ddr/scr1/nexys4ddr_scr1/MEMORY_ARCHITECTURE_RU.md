# Память в архитектуре SCR1 на Nexys4 DDR

## Краткий ответ

В данной реализации используются три настоящих области памяти:

| Адресный диапазон | Размер | Назначение | Физическая реализация |
|---|---:|---|---|
| `0x00000000`–`0x07FFFFFF` | 128 МиБ | Основная память для программ и данных | Внешняя микросхема DDR2 на плате, контроллер MIG 7 Series |
| `0xF0000000`–`0xF000FFFF` | 64 КиБ | TCM — быстрая локальная память процессора | Внутренняя Block RAM кристалла Artix-7, описанная RTL-кодом |
| `0xFFFF0000`–`0xFFFFFFFF` | 64 КиБ | Boot SRAM/OCRAM с первичным загрузчиком SCBL | Внутренняя Block RAM Artix-7, созданная IP `Block Memory Generator` |

Кроме памяти, в адресном пространстве есть MMIO-области: UART, идентификаторы SoC/сборки, регистр частоты и таймер. Это регистры периферии, а не RAM.

Важно: память загрузчика начинается с `0xFFFF0000`. Адрес `0xFFFFFF00` — это **точка старта процессора после сброса**, расположенная внутри этой памяти.

## Общая схема

```text
                         +------------------------+
                         |       SCR1 core        |
                         |                        |
                         |  instruction  data     |
                         +-------+--------+-------+
                                 |        |
                    +------------+--------+------------+
                    |   внутренние роутеры imem/dmem   |
                    +------+---------------------+------+
                           |                     |
             адрес F000_xxxx                     | остальные адреса
                           |                     |
                    +------v------+       +------v------+
                    | TCM, 64 КиБ |       | AXI imem /  |
                    | FPGA BRAM   |       | AXI dmem    |
                    +-------------+       +------+------+ 
                                                  |
                                           SmartConnect
                             +--------------------+-------------------+
                             |                    |                   |
                      Boot BRAM/OCRAM          UART/MMIO          DDR2 MIG
                      FFFF0000–FFFFFFFF                           00000000–
                      64 КиБ FPGA BRAM                             07FFFFFF
```

TCM обслуживается внутри обвязки SCR1 и не проходит через внешний AXI SmartConnect. Обращения к DDR2, boot BRAM и периферии проходят через AXI.

## 1. Внешняя DDR2

Диапазон DDR2:

```text
0x00000000 ... 0x07FFFFFF = 0x08000000 байт = 128 МиБ
```

Это не память внутри FPGA. Данные физически находятся во внешней микросхеме DDR2 SDRAM, установленной на Nexys4 DDR. FPGA содержит только логику доступа к ней.

Путь транзакции выглядит так:

```text
SCR1 imem/dmem
    -> AXI SmartConnect, выход M03_AXI
    -> AXI Clock Converter
    -> MIG 7 Series
    -> физические сигналы DDR2
    -> микросхема DDR2 на плате
```

MIG выполняет инициализацию и калибровку DDR2, формирует команды памяти и соблюдает необходимые временные параметры. Clock Converter нужен потому, что процессорная AXI-шина и пользовательский интерфейс MIG работают в разных тактовых доменах.

Обе AXI-шины процессора могут обращаться к DDR2:

- `axi_imem` читает оттуда инструкции;
- `axi_dmem` читает и записывает данные.

Соответствующие адресные сегменты заданы в [`nexys4ddr_scr1.tcl`](../nexys4ddr_scr1.tcl):

```tcl
create_bd_addr_seg -range 0x08000000 -offset 0x00000000 \
    [get_bd_addr_spaces axi_dmem] ...
create_bd_addr_seg -range 0x08000000 -offset 0x00000000 \
    [get_bd_addr_spaces axi_imem] ...
```

После конфигурирования FPGA DDR2 не содержит загрузчик автоматически: это энергозависимая память, которую сначала должен инициализировать MIG, а затем заполнить программа, например SCBL через UART.

## 2. TCM

TCM — Tightly-Coupled Memory, то есть память, непосредственно подключённая к процессорному тракту:

```text
0xF0000000 ... 0xF000FFFF = 64 КиБ
```

Она предназначена для небольших программ и данных, которым нужен простой и детерминированный доступ без DDR-контроллера, калибровки и AXI SmartConnect.

Адрес TCM задаётся маской и шаблоном в [`scr1_arch_custom.svh`](../src/scr1_arch_custom.svh):

```systemverilog
SCR1_TCM_ADDR_MASK    = 'hFFFF0000;
SCR1_TCM_ADDR_PATTERN = 'hF0000000;
```

Маска `0xFFFF0000` оставляет изменяемыми младшие 16 бит адреса, поэтому фактический размер области равен `2^16 = 65536` байт.

TCM описана в RTL-модулях [`scr1_tcm.sv`](../../../../scr1/src/top/scr1_tcm.sv) и [`scr1_dp_memory.sv`](../../../../scr1/src/top/scr1_dp_memory.sv). Массив памяти синтезатор Vivado отображает на аппаратные BRAM-примитивы FPGA. Поэтому TCM — не DDR и не отдельная микросхема SRAM на плате: она собирается из встроенных блоков памяти Artix-7.

У TCM есть пути для инструкций и данных. Внутренние роутеры SCR1 распознают диапазон `0xF000xxxx` и направляют запрос непосредственно в TCM, до преобразования native-интерфейса SCR1 в AXI.

Практическое следствие для отладки: загрузка теста в TCM видна на интерфейсе ядра/TCM, но не на выходах внешнего SmartConnect.

## 3. Boot SRAM/OCRAM

Верхние 64 КиБ адресного пространства отданы памяти загрузчика:

```text
0xFFFF0000 ... 0xFFFFFFFF = 64 КиБ
```

В документации проекта она называется SRAM или on-chip SRAM. Физически это также **Block RAM внутри FPGA**, а не внешняя SRAM и не DDR2.

В block design память создаётся IP-ядром Xilinx `blk_mem_gen_0` типа `True Dual Port RAM`:

```tcl
set blk_mem_gen_0 [create_bd_cell -type ip \
    -vlnv xilinx.com:ip:blk_mem_gen:8.4 blk_mem_gen_0]
```

Её подключение:

```text
axi_imem/axi_dmem
    -> SmartConnect M00_AXI
    -> AXI BRAM Controller
    -> BRAM_PORTA и BRAM_PORTB
    -> blk_mem_gen_0
    -> RAMB36/RAMB18 внутри Artix-7
```

Для `axi_imem` и `axi_dmem` в [`nexys4ddr_scr1.tcl`](../nexys4ddr_scr1.tcl) создан один и тот же 64-КиБ сегмент с базой `0xFFFF0000`. Поэтому процессор может как выбирать из этой памяти инструкции, так и обращаться к ней как к данным.

## Откуда берётся блочная память

Block RAM — аппаратный ресурс самого кристалла FPGA. Кристалл Artix-7 уже содержит множество отдельных блоков двухпортовой SRAM. Они существуют наряду с LUT, триггерами, DSP и другими ресурсами FPGA.

Vivado использует эти блоки двумя способами:

1. Явно — через IP `Block Memory Generator`, как для boot SRAM `blk_mem_gen_0`.
2. Через вывод памяти из RTL-описания, как для TCM в `scr1_dp_memory.sv`.

При загрузке bitstream конфигурируется не только соединение логических элементов, но и начальное содержимое BRAM. После выключения питания это содержимое исчезает; при следующем программировании платы оно восстанавливается из bitstream.

Таким образом, в проекте есть две независимые 64-КиБ памяти на BRAM:

- внутренняя TCM SCR1 по адресу `0xF0000000`;
- AXI-доступная boot SRAM по адресу `0xFFFF0000`.

Это разные экземпляры памяти и разные адресные области.

## Почему фигурирует адрес `0xFFFFFF00`

В память не «пишут именно с `0xFFFFFF00`». Здесь смешиваются три разных понятия:

- `0xFFFF0000` — начало 64-КиБ boot SRAM;
- `0xFFFFFF00` — reset vector, то есть первое значение PC после сброса;
- адрес назначения, введённый пользователем в SCBL, — место, куда загрузчик записывает принимаемый по UART тест, обычно `0xF0000000` для TCM или `0x00000000` для DDR2.

Reset vector платформы задан в [`scr1_arch_custom.svh`](../src/scr1_arch_custom.svh):

```systemverilog
SCR1_ARCH_RST_VECTOR = 'hFFFFFF00;
```

Адрес находится в последних 256 байтах boot SRAM:

```text
начало boot SRAM                        reset vector
0xFFFF0000 ---------------------------------- 0xFFFFFF00 ---- 0xFFFFFFFF
```

Такое размещение согласовано с linker script загрузчика [`scbl.ld`](../../../../sw/sc-bl/common/scbl.ld):

```ld
MEMORY {
  RAM (rwx) : ORIGIN = 0-64K, LENGTH = 64K
}

.startup ORIGIN(RAM) + LENGTH(RAM) - 0x800 : { ... }
.vectors ORIGIN(RAM) + LENGTH(RAM) - 0x100 : { ... }
```

В 32-битном адресном пространстве выражение `0 - 64K` соответствует `0xFFFF0000`. Поэтому:

```text
ORIGIN(RAM)                         = 0xFFFF0000
ORIGIN + LENGTH - 0x800             = 0xFFFFF800  (.startup)
ORIGIN + LENGTH - 0x100             = 0xFFFFFF00  (.vectors)
```

После сброса SCR1 устанавливает PC в `0xFFFFFF00`, выбирает инструкцию из секции `.vectors`, а та передаёт управление стартовому коду и далее `main()` загрузчика.

Выбор верхней части адресного пространства удобен тем, что:

- загрузочная BRAM отделена от основной DDR2 с базой `0x00000000`;
- reset vector попадает внутрь небольшой локальной памяти, доступной сразу после конфигурации FPGA;
- процессор может стартовать, не ожидая заполнения DDR2 прикладной программой.

## Как загрузчик попадает в BRAM

Загрузчик SCBL не приходит по UART. Он встраивается в конфигурацию FPGA на этапе подготовки bitstream.

Последовательность следующая:

1. Исходники из `sw/sc-bl` компилируются и линкуются по `scbl.ld`.
2. Из результата формируется файл `scbl.mem` с содержимым памяти загрузчика.
3. Vivado создаёт обычный реализованный bitstream `nexys4ddr_scr1.bit`.
4. Скрипт [`mem_update.tcl`](../mem_update.tcl) находит в реализованном дизайне память `blk_mem_gen_0`.
5. Процедура из [`write_mmi.tcl`](../write_mmi.tcl) создаёт `.mmi` — описание того, какие физические BRAM соответствуют адресам логической памяти.
6. Утилита `updatemem` помещает данные из `scbl.mem` в INIT-содержимое соответствующих BRAM внутри bitstream.
7. Получается обновлённый файл `nexys4ddr_scr1_new.bit`.
8. При программировании FPGA этим файлом одновременно создаётся вся схема и инициализируется boot BRAM загрузчиком.

Ключевая команда из `mem_update.tcl`:

```tcl
exec updatemem -force \
    --meminfo $mmi_file \
    --data $mem_file \
    --bit $bit_file \
    --proc dummy \
    --out $new_bit_file
```

То есть загрузчик уже находится в BRAM к моменту снятия reset. Процессору не требуется сначала получать его из DDR2, UART или отдельной flash-памяти.

## Что происходит после включения

```text
Программирование FPGA файлом nexys4ddr_scr1_new.bit
    -> конфигурация логики и начального содержимого boot BRAM
    -> снятие reset
    -> PC = 0xFFFFFF00
    -> выполнение .vectors и .startup из boot BRAM
    -> запуск SCBL
    -> SCBL принимает команду и тест по UART/XMODEM
    -> SCBL пишет тест по выбранному адресу:
         0xF0000000 — TCM
         0x00000000 — DDR2
    -> команда запуска передаёт управление на адрес теста
```

Иными словами, существуют два независимых этапа загрузки:

1. **SCBL → boot BRAM**: выполняется инструментами Vivado/`updatemem` при создании bitstream.
2. **Тест → TCM или DDR2**: выполняется уже работающим SCBL через UART/XMODEM.

## Важные замечания

### TCM фактически имеет 64 КиБ

В отдельных заголовочных файлах SDK может встречаться другое значение, однако для данного FPGA-дизайна маска `0xFFFF0000`, linker scripts и RTL задают область `0xF0000000`–`0xF000FFFF`, то есть 64 КиБ. При анализе именно этой сборки следует считать фактическим размером 64 КиБ.

### MMIO — не память для программы

Адреса UART и служебных блоков вида `0xFFxxxxxx` участвуют в общей карте адресов, но обращение к ним приводит к чтению или записи регистров IP-блоков. Хранить там код или массивы нельзя.

### Кэш-память

В рассматриваемой конфигурации активной instruction/data cache перед DDR2 нет. Поэтому DDR2 остаётся внешней памятью с существенно более сложным и менее детерминированным доступом, чем TCM.
