# Разбор загрузчика `scbl.c`

`scbl.c` — это минимальный first-stage bootloader/монитор. Он не умеет разбирать ELF, выбирать секции или самостоятельно определять точку входа. Его основная схема очень простая:

```text
reset → запуск SCBL из BRAM → UART-командная строка
                               │
                               ├─ 1: принять сырой .bin по XMODEM
                               │     и записать начиная с указанного адреса
                               │
                               └─ g: выполнить jalr на указанный адрес
```

Главный файл: [scbl.c](D:/scr1-sdk/sw/sc-bl/src/scbl.c:1).

## Где и как стартует сам SCBL

SCBL находится не в TCM и не в DDR, а в 64-КиБ on-chip BRAM по адресу:

```text
0xFFFF0000 ... 0xFFFFFFFF
```

Линкер описывает это несколько необычно:

```ld
RAM (rwx) : ORIGIN = 0-64K, LENGTH = 64K
```

Для 32-битного адресного пространства это превращается в `0xFFFF0000`. Стартовый код помещён в `0xFFFFF800`, а таблица векторов — в `0xFFFFFF00`: [scbl.ld](D:/scr1-sdk/sw/sc-bl/common/scbl.ld:13).

Процессор действительно настроен стартовать с `0xFFFFFF00`: [scr1_arch_custom.svh](D:/scr1-sdk/fpga/nexys4ddr/scr1/src/scr1_arch_custom.svh:31).

Содержимое `scbl.mem` встраивается в BRAM уже после реализации FPGA через `updatemem`: [updatemem.log](D:/scr1-sdk/fpga/nexys4ddr/scr1/updatemem.log:8).

До входа в `main()` выполняется [startup.S](D:/scr1-sdk/sw/sc-bl/src/startup.S:64):

- настраивается `mstatus`;
- устанавливается `mtvec` на обработчик ловушек SCBL;
- устанавливается `gp`;
- вторичные hart переводятся в halt;
- очищается `.bss`;
- настраиваются стек и TLS;
- вызывается `main()`.

То есть `scbl.c` не является самым первым выполняемым кодом — перед ним есть полноценный CRT-стартап.

## Что делает `main()`

Главный цикл начинается в [scbl.c](D:/scr1-sdk/sw/sc-bl/src/scbl.c:303).

### 1. Инициализация таймера

```c
scr_rtc_init();
```

Сбрасывается `mtime`, устанавливается делитель до 1 МГц и запускается машинный таймер. Он нужен главным образом для задержек и таймаутов XMODEM: [rtc.h](D:/scr1-sdk/sw/sc-bl/common/rtc.h:79).

### 2. Инициализация UART

```c
sc1f_uart_init();
```

UART работает как 16550:

- 8 бит данных;
- без чётности;
- один стоп-бит;
- FIFO включён;
- UART-прерывания выключены;
- весь обмен выполняется polling-циклами.

Реализация находится в [uart.c](D:/scr1-sdk/sw/sc-bl/common/uart.c:12).

При частоте 30 МГц получается делитель 16. Фактическая скорость линии будет около `117187.5 baud`, а не ровно `115200`; ошибка около 1,73%. Для обычного UART это чаще всего допустимо, но при декодировании ILA или внешним логическим анализатором это стоит учитывать.

### 3. Инициализация LED

```c
sc1f_leds_init();
```

В платформенном заголовке Nexys4DDR нет `PLF_PINLED_ADDR` и `PLF_HEXLED_ADDR`, поэтому для данной платформы большая часть LED-кода фактически превращается в пустые макросы. Анимация из `indication_tasks()` здесь практически не работает.

### 4. Печать заголовка и карты памяти

Загрузчик выводит версию, ISA, частоты, идентификаторы FPGA и карту памяти:

```c
show_header();
hwinfo();
usage();
```

`hwinfo()` читает:

- `misa`;
- `mimpid`;
- system/build ID;
- частоту CPU;
- статическую карту памяти из платформенного заголовка.

Важно: напечатанная карта памяти не является результатом аппаратного обнаружения. Это просто массив констант из [plf_nexys4ddr_scr1.h](D:/scr1-sdk/sw/sc-bl/src/plf_nexys4ddr_scr1.h:24).

### 5. Командный цикл

После инициализации появляется приглашение:

```text
:
```

Загрузчик ждёт один символ. Пока символа нет, он вызывает задачи индикации.

Таблица команд находится в [scbl.c](D:/scr1-sdk/sw/sc-bl/src/scbl.c:192):

| Команда | Назначение |
|---|---|
| `1` | принять файл по XMODEM |
| `g` | передать управление по адресу |
| `d` | вывести 128 байт памяти |
| `m` | записать 32-битное слово |
| `i` | вывести информацию о платформе |
| пробел | повторно показать список команд |

## Команда `1`: загрузка XMODEM

Команда вызывает:

```c
xmodem_receive((uint8_t*)arg, MAX_XMODEM_RX_LEN);
```

где:

```c
#define MAX_XMODEM_RX_LEN PLF_MEM_SIZE
```

Для Nexys4DDR это 128 МиБ: [scbl.c](D:/scr1-sdk/sw/sc-bl/src/scbl.c:21).

После ввода адреса загрузчик начинает передавать символ `C`. Это означает: «готов принимать XMODEM с CRC16».

Последовательность выглядит примерно так:

```text
Плата → PC:  'C'

PC → плата:
    SOH/STX
    номер пакета
    инверсия номера
    128 или 1024 байта данных
    CRC16

Плата → PC:
    ACK — пакет принят
    NAK — ошибка

PC → плата:
    EOT

Плата → PC:
    ACK
```

Реализация находится в [xmodem.c](D:/scr1-sdk/sw/sc-bl/src/xmodem.c:61).

Поддерживаются:

- `SOH` — пакет с 128 байтами;
- `STX` — пакет с 1024 байтами;
- CRC16-CCITT;
- повтор предыдущего пакета;
- `CAN CAN` для отмены;
- `EOT` для завершения.

Данные записываются последовательно:

```c
buf[rxsize] = c;
```

То есть это сырой побайтовый образ. Никакого ELF-разбора нет.

Отправлять нужно `.bin`, созданный примерно так:

```text
objcopy -O binary program.elf program.bin
```

Если отправить `.elf`, в память попадёт ELF-заголовок, и переход на начало образа почти наверняка закончится исключением.

### Размер после XMODEM

Загрузчик возвращает не точный размер файла, а сумму размеров XMODEM-пакетов. Например, файл размером 100 байт будет принят как 128 байт.

Последний пакет дополняется передатчиком, обычно байтами `0x1A`. Загрузчик padding не удаляет, хотя константа `CTRLZ` определена.

Поэтому сообщение:

```text
Xmodem successfully received 128 bytes
```

для 100-байтового файла нормально.

### Отмена загрузки

После запуска команды `1` обычного таймаута возврата в командную строку нет. Получатель может передавать `C` бесконечно. Обработка `Ctrl-C` в коде закомментирована.

Для выхода нужны два байта `CAN` (`0x18`) либо reset платы.

## Команда `g`: передача управления

Реализация:

```c
uart_putc('\n');
sc1f_uart_tx_flush();
rtc_delay_us(20000);
asm volatile ("jalr %0" :: "r"(arg));
```

См. [scbl.c](D:/scr1-sdk/sw/sc-bl/src/scbl.c:152).

Это означает:

1. дождаться физического опустошения UART TX;
2. подождать ещё 20 мс;
3. выполнить `jalr` на введённый адрес.

Это не reset процессора. Сохраняются:

- машинный режим;
- `mtvec` SCBL;
- состояние CSR;
- значение `mscratch`;
- состояние периферии;
- часть регистров и текущая программная среда.

Фактически загруженная программа вызывается как подпрограмма. `jalr` сохраняет адрес возврата в `ra`. Но возврат программы в SCBL ненадёжен: она могла изменить стек, `gp`, CSR и другие регистры.

Загружаемый образ должен содержать собственный startup-код, который:

- установит `sp` и `gp`;
- очистит `.bss`;
- при необходимости установит `mtvec`;
- вызовет `main()`.

### Адрес загрузки и адрес запуска могут различаться

Для SDK-тестов документация прямо указывает:

```text
load: 0xF0000000
run:  0xF0000200
```

См. [tests/README.md](D:/scr1-sdk/sw/tests/README.md:60).

Образ загружается с начала TCM, но `_start` расположен по смещению `0x200`. Поэтому нельзя автоматически выполнять `g` с тем же адресом, который использовался в команде `1`.

Точку входа лучше всегда проверять в ELF:

```text
riscv32-unknown-elf-readelf -h program.elf
```

и брать значение `Entry point address`.

Типичная последовательность для SDK-теста:

```text
: 1
addr: f0000000

[передать test.bin по XMODEM]

: g
addr: f0000200
```

## Команды просмотра и изменения памяти

### `d` — dump

Команда выводит 128 байт начиная с указанного адреса:

```text
F0000000:  00 01 02 03 ... |....|
```

При последующих нажатиях Enter адрес автоматически увеличивается на 128.

Есть важная особенность: каждый байт читается дважды:

- первый раз для шестнадцатеричного представления;
- второй раз для ASCII-представления.

Для обычной RAM это безвредно. Для MMIO это опасно: регистр может иметь read-to-clear семантику или чтение может извлекать байт из FIFO. Поэтому `d ff010000` способен влиять на UART и показывать разные значения в HEX- и ASCII-частях.

Неверный или неотображённый адрес приводит к trap, после которого SCBL останавливает hart. Автоматического возврата к приглашению нет: [trap.c](D:/scr1-sdk/sw/sc-bl/src/trap.c:60).

### `m` — modify

Команда принимает число и записывает:

```c
*(volatile unsigned long*)arg = val;
```

На RV32 `unsigned long` имеет размер 4 байта. Адрес должен быть выровнен на 4.

Повторное нажатие Enter увеличивает адрес на 4.

Запись в MMIO выполняется полным 32-битным словом. Она может менять несколько битов периферии сразу и иметь необратимые побочные эффекты.

### Повтор предыдущей команды

Enter повторяет только команды с `SCBL_CMD_ARG_AUTOINC`, то есть `d` и `m`.

Есть небольшой сюрприз: неизвестная команда не очищает `prev_cmd`. Поэтому после ошибочного символа следующий Enter способен продолжить старый dump/modify.

## Минимальный SBI-подобный сервис

После `g` обработчик ловушек SCBL остаётся установленным в `mtvec`. Он поддерживает несколько старых SBI-подобных вызовов:

- console putchar;
- console getchar;
- shutdown.

См. [trap.c](D:/scr1-sdk/sw/sc-bl/src/trap.c:93).

`shutdown` вызывает `_start` загрузчика и возвращает систему к SCBL. Это программный перезапуск загрузчика, а не аппаратный reset FPGA.

Если загруженная программа устанавливает собственный `mtvec`, эти сервисы SCBL перестают использоваться.

## Как это выглядит на аппаратных интерфейсах

Для отладки ILA полезно сопоставить C-код с транзакциями.

Пока SCBL ждёт команду:

```c
sc1f_uart_getch_nowait()
```

процессор практически непрерывно читает UART status:

```text
0xFF010014
```

Поэтому на `axi_dmem` будет много однотипных чтений этого адреса.

При появлении байта:

```text
read 0xFF010000
```

Во время XMODEM:

- чтения `0xFF010014` — проверка RX ready;
- чтения `0xFF010000` — принятые UART-байты;
- записи `0xFF010000` — `C`, `ACK`, `NAK`, `CAN`;
- записи в адрес загрузки — полезные данные файла.

Есть важное отличие для TCM. Диапазон `0xF0000000...0xF000FFFF` маршрутизируется во внутренний TCM SCR1 до AXI-преобразователя: [scr1_top_axi.sv](D:/scr1-sdk/scr1/src/top/scr1_top_axi.sv:362).

Поэтому:

- загрузка в DDR `0x00000000...` видна на внешнем `axi_dmem` и SmartConnect;
- загрузка в TCM `0xF0000000...` на внешнем SmartConnect не видна;
- исполнение из DDR видно на `axi_imem`;
- исполнение из TCM не выходит на внешний `axi_imem`.

Это особенно важно, если вы загружаете стандартные SDK-тесты в TCM и ожидаете увидеть записи на SmartConnect: их там не будет.

## Главные проблемные места

### 1. Неправильное ограничение размера загрузки

Самая серьёзная проблема — `MAX_XMODEM_RX_LEN` всегда равен размеру DDR, независимо от введённого адреса.

Если указать TCM `0xF0000000`, загрузчик всё равно считает, что разрешено принять 128 МиБ. Но реальный TCM здесь имеет только 64 КиБ:

```text
0xF0000000 ... 0xF000FFFF
```

Это следует из маски `0xFFFF0000`: [scr1_arch_custom.svh](D:/scr1-sdk/fpga/nexys4ddr/scr1/src/scr1_arch_custom.svh:34).

Большой файл выйдет за пределы TCM и приведёт к fault. Аналогично загрузка в середину DDR всё равно получает лимит 128 МиБ вместо «размер DDR минус смещение».

### 2. Несоответствие размера TCM

Платформенный заголовок SCBL объявляет:

```c
#define PLF_TCM_SIZE (128*1024)
```

Но:

- аппаратная маска задаёт 64 КиБ;
- тестовый линкер использует 64 КиБ: [tcm.ld](D:/scr1-sdk/sw/tests/tests_common/tcm.ld:14);
- README также говорит 64 КиБ.

Из-за этого команда `i` печатает неверную карту памяти. Это стоит исправить в первую очередь.

### 3. XMODEM неверно обрабатывает превышение `destsz`

В [xmodem.c](D:/scr1-sdk/sw/sc-bl/src/xmodem.c:133) при превышении лимита запись просто пропускается:

```c
if (totalbytes + pktsize <= destsz)
    buf[rxsize] = c;
```

Но после успешного CRC пакет всё равно подтверждается `ACK`, а `totalbytes` увеличивается. Передатчик может получить успешное завершение, хотя часть данных не была записана.

Правильное поведение — отправить `CAN`/ошибку сразу при пакете, который не помещается.

### 4. Адрес полностью доверенный

Нет проверки:

- принадлежит ли адрес RAM;
- достаточно ли места до конца области;
- не попадает ли загрузка в SCBL;
- не попадает ли адрес в MMIO;
- выровнена ли точка входа;
- является ли адрес исполняемым.

Пустой ввод адреса возвращает ноль. Для DDR это может оказаться допустимым адресом, но для `g` это не обязательно точка входа.

### 5. Нет `fence.i`

После записи кода через data-интерфейс перед `jalr` нет:

```asm
fence rw, rw
fence.i
```

На текущей простой конфигурации без I-cache переход обычно работает. Но архитектурно после изменения памяти данных и последующего исполнения из неё требуется синхронизация instruction fetch. При добавлении I-cache это станет особенно важным.

### 6. Нет проверки всего образа

XMODEM проверяет CRC каждого пакета, но загрузчик не проверяет:

- magic;
- полный размер;
- адрес загрузки;
- entry point;
- CRC всего образа;
- версию/совместимость;
- подпись.

Для лабораторной платы это допустимо, но любой успешно принятый набор байтов затем можно выполнить.

### 7. DDR readiness не проверяется

SCBL не ожидает `ddr2_init_complete`. В топе этот сигнал только выведен на `LED[0]`: [nexys4ddr_scr1.sv](D:/scr1-sdk/fpga/nexys4ddr/scr1/src/nexys4ddr_scr1.sv:516).

Если начать XMODEM-запись в DDR слишком рано, первая AXI-запись может надолго остановить процессор. За это время RX FIFO UART способен переполниться. Для загрузки в DDR разумно сначала дождаться LED0.

### 8. Особенности восстановления XMODEM

В реализации есть несколько слабых мест:

- байты пишутся в память до проверки CRC пакета;
- после `NAK` вызывается `flushinput()`, способный съесть начало быстрой повторной передачи;
- `MAXRETRANS` уменьшается только при повторе предыдущего пакета, а не при большинстве CRC/timeout ошибок;
- на переходе номера пакета `255 → 0` повтор пакета 255 распознаётся неправильно;
- после EOT `flushinput()` ждёт длительный период тишины, поэтому сообщение об успехе появляется с заметной задержкой.

## Что я бы исправлял в первую очередь

1. Исправил `PLF_TCM_SIZE` на 64 КиБ.
2. Добавил определение области по адресу и вычисление реального доступного размера.
3. Запрещал бы запись в OCRAM SCBL и MMIO.
4. Принимаемый XMODEM-пакет сначала сохранял бы во временный буфер, затем проверял CRC и только после этого копировал в целевую память.
5. Добавил бы заголовок образа: magic, размер, load address, entry point и CRC32.
6. Перед `jalr` выполнял бы `fence` и `fence.i`.
7. Для DDR ожидал бы `ddr2_init_complete`.
8. Добавил бы нормальную отмену XMODEM по Ctrl-C и общий лимит повторов.
9. Для `g` использовал бы явно объявленный assembly trampoline вместо небезопасного inline `jalr`.

В текущем виде SCBL хорошо подходит как простой стендовый монитор: загрузить доверенный небольшой `.bin`, посмотреть память и запустить. Но он предполагает, что пользователь уже знает точные load/entry адреса и не ошибается с размером или областью памяти.
